## Summary

Cette phase ajoute la gestion des invitations Clerk directement depuis `/parametres/organisation`, sans passer par le dashboard Clerk. Le travail est divisé en trois parties : (1) trois nouvelles server actions dans `src/actions/settings.ts` utilisant le Clerk Backend API, (2) un composant dialog d'invitation (`InviteDialog`), et (3) une section "Invitations en attente" affichée sous le `TeamTable` existant.

La découverte critique de cette recherche est que **`clerkClient` dans `@clerk/nextjs` v7 est une fonction async** — il faut l'appeler `await clerkClient()` puis chaîner `.organizations`, contrairement à la syntaxe v5/v6 où c'était un objet direct. Aucune donnée n'est stockée en Prisma : les invitations vivent entièrement dans Clerk. La page sera un Server Component qui lit les invitations au chargement ; les mutations (créer/révoquer) passent par server actions + `revalidatePath`.

**Primary recommendation:** Appeler `(await clerkClient()).organizations.createOrganizationInvitation(...)` dans les server actions. La liste des invitations est récupérée côté serveur dans la page (Server Component), filtrée sur `status: ['pending']`.

### `OrganizationMembershipRole` et rôles Clerk

Le type `OrganizationMembershipRole` est un alias de `OrganizationCustomRoleKey` (string).
Dans Clerk, les rôles custom sont préfixés `org:` par convention (ex: `org:admin`, `org:technician`).

**ATTENTION:** L'enum `MemberRole` Prisma (`admin`, `manager`, etc.) est sans préfixe. La correspondance avec les rôles Clerk dépend de la configuration de l'organisation Clerk.

**Point à vérifier lors de l'implémentation :** Utiliser les valeurs de `MemberRole` Prisma directement ou les préfixer `org:` selon ce qui est configuré dans le dashboard Clerk. Regarder les valeurs de `role` retournées par `getOrganizationMembershipList` pour confirmer.

### Anti-Patterns à éviter

- **Ne pas appeler `clerkClient.organizations.*` directement** — c'est une fonction async en v7, toujours `await clerkClient()` d'abord.
- **Ne pas stocker les invitations en Prisma** — elles vivent dans Clerk. `revalidatePath` suffit à rafraîchir la liste.
- **Ne pas passer l'`organizationId` Prisma à Clerk** — Clerk utilise `orgId` (le `clerkId`), pas l'UUID Prisma.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Envoi d'email d'invitation | Email custom + token | `clerkClient().organizations.createOrganizationInvitation()` | Clerk gère l'email, le token, l'expiration, la page d'acceptation |
| Gestion statut invitation | Table Prisma `invitations` | `getOrganizationInvitationList({ status: ['pending'] })` | Clerk maintient le statut (accepted, revoked, expired) automatiquement |
| Révocation | Soft-delete Prisma | `revokeOrganizationInvitation()` | Clerk invalide le token et met à jour le statut |

## Common Pitfalls

### Pitfall 1: `clerkClient` appelé sans `await` (v7 breaking change)

**What goes wrong:** `clerkClient.organizations.createOrganizationInvitation(...)` → TypeError: clerkClient.organizations is undefined
**Why it happens:** En v7, `clerkClient` est `async () => ClerkClient`, pas un objet singleton.
**How to avoid:** Toujours `const client = await clerkClient()` puis `client.organizations.*`
**Warning signs:** TypeScript signale que `clerkClient` est `() => Promise<ClerkClient>`, pas `ClerkClient`

[VERIFIED: node_modules/@clerk/nextjs/dist/cjs/server/clerkClient.js ligne 30]

### Pitfall 2: Mauvais `organizationId` passé à Clerk

**What goes wrong:** Passer l'UUID Prisma `org.id` au lieu du Clerk org ID `orgId` → 404 de l'API Clerk
**Why it happens:** Le projet a deux IDs d'organisation : l'UUID Prisma et le Clerk `orgId` (format `org_xxx`)
**How to avoid:** Utiliser `orgId` issu de `auth()` directement pour tous les appels Clerk Backend API
**Warning signs:** Erreur 404 ou "Organization not found" de Clerk

### Pitfall 3: Format du rôle Clerk (préfixe `org:`)

**What goes wrong:** Passer `'technician'` au lieu de `'org:technician'` → erreur Clerk "role not found"
**Why it happens:** Clerk custom roles ont souvent le préfixe `org:` selon la configuration
**How to avoid:** Vérifier la valeur exacte du champ `role` sur un membre existant via `getOrganizationMembershipList()` en développement avant de coder en dur les valeurs
**Warning signs:** Erreur 422 de Clerk lors de `createOrganizationInvitation`

[ASSUMED: les rôles Clerk sont préfixés `org:` — à confirmer en dev]

### Pitfall 4: `createdAt` Clerk est un timestamp ms, pas une Date

**What goes wrong:** `new Date(invitation.createdAt)` fonctionne mais `invitation.createdAt.toLocaleDateString()` crash
**Why it happens:** `OrganizationInvitation.createdAt` est `number` (ms), pas un objet `Date`
**How to avoid:** `new Date(invitation.createdAt)` pour formatter avec `Intl.DateTimeFormat`

[VERIFIED: node_modules/@clerk/backend/dist/api/resources/OrganizationInvitation.d.ts]

### Pitfall 5: La vitest.setup.ts mocke `@clerk/nextjs/server` sans `clerkClient`

**What goes wrong:** Tests du server action `inviteMember` → `clerkClient is not a function` ou `clerkClient` non mocké
**Why it happens:** `vitest.setup.ts` ne mocke que `auth` et `currentUser`, pas `clerkClient`
**How to avoid:** Dans le test file, ajouter `vi.mock('@clerk/nextjs/server', ...)` avec `clerkClient` mocké
**Warning signs:** Test passe globalement mais appelle le vrai Clerk SDK

[VERIFIED: src/vitest.setup.ts]

## Open Questions

1. **Format exact des rôles Clerk (préfixe `org:` ou non)**
   - What we know: `OrganizationMembershipRole` est un string alias; Clerk custom roles utilisent souvent `org:` par convention
   - What's unclear: Configuration spécifique de cette instance Clerk Korvia
   - Recommendation: Vérifier en dev via log ou Clerk dashboard avant de coder le mappage

2. **La `getAdminOrg()` existante retourne `organizationId` Prisma mais les appels Clerk nécessitent `orgId` Clerk**
   - What we know: `getAdminOrg()` retourne `org.id` (UUID Prisma) après vérification des droits
   - What's unclear: Si `getAdminOrg()` doit être modifiée ou si on appelle `auth()` séparément pour avoir `orgId`
   - Recommendation: Appeler `auth()` en haut de chaque nouvelle action pour obtenir `orgId` Clerk, puis appeler `getAdminOrg()` seulement pour la vérification des droits. Ou étendre `getAdminOrg()` pour retourner aussi `{ orgId, organizationId }`.
